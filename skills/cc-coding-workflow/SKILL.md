# Taro CC 编程工作流

## 核心原则

> **先评估，后行动。复杂任务交给专业工具，简单任务自己搞定。**

## 任务评估标准

收到编程任务时，从以下维度评估：

| 维度 | 简单任务（我自己写） | 复杂任务（spawn Kimi Code） |
|------|---------------------|---------------------------|
| **文件数量** | 单文件或 2-3 个文件 | 多文件、跨目录修改 |
| **代码量** | < 100 行新增/修改 | > 100 行或大规模重构 |
| **架构影响** | 局部改动，不影响整体 | 涉及架构、接口变更 |
| **调试需求** | 无需运行测试 | 需要测试、调试、验证 |
| **上下文依赖** | 需求清晰，无需探索 | 需要理解现有代码库 |
| **技术栈** | 熟悉的语言/框架 | 陌生技术栈或复杂依赖 |

## 决策流程

```
用户提出编程需求
    ↓
【快速评估】满足以下任一条件 → 使用 Kimi Code
    ├─ 涉及多个文件的修改
    ├─ 需要测试/验证流程
    ├─ 需要理解陌生代码库
    ├─ 代码量预计 > 100 行
    ├─ 涉及构建/部署流程
    └─ 用户明确要求"完整实现"
    ↓
是 → spawn Kimi Code 子代理
否 → 我自己直接写
```

## 两种执行模式

### 模式 A：自己直接写（简单任务）

适用场景：
- 单文件脚本
- 小功能实现
- 快速修复
- 配置调整
- 简单的代码审查

流程：
1. 确认需求
2. 直接编写代码
3. 简要说明改动
4. 交付结果

### 模式 B：Spawn Kimi Code（复杂任务）

适用场景：
- 完整功能开发
- 多文件修改
- Bug 排查修复
- 项目初始化/脚手架
- 代码重构
- 需要测试覆盖

流程：

```
1. 需求澄清
   └─ 确认功能需求、约束条件、验收标准

2. Spawn Kimi Code
   └─ sessions_spawn 创建子代理
   └─ 传递完整上下文（项目路径、需求描述、约束）

3. 监控执行
   └─ 子代理自主执行：读代码 → 规划 → 实现 → 测试
   └─ 我保持关注，必要时介入

4. 结果验收
   └─ Review 子代理的修改
   └─ 验证是否满足需求
   └─ 必要时要求调整

5. 整合汇报
   └─ 向用户总结完成的工作
   └─ 说明关键改动和注意事项
```

## Spawn Kimi Code 的规范

### 启动参数

```javascript
sessions_spawn({
  runtime: "acp",
  agentId: "kimi-code",  // 或其他 coding agent
  task: "详细任务描述",
  cwd: "项目绝对路径",
  mode: "session",  // 复杂任务用 session 保持上下文
  thread: true
})
```

### 任务描述模板

```
【任务】实现 XXX 功能
【项目路径】/path/to/project
【需求详情】
- 功能点 1
- 功能点 2
- 约束条件

【验收标准】
- [ ] 功能 A 可用
- [ ] 通过测试 B
- [ ] 代码符合项目风格

【注意事项】
- 依赖版本 X.Y.Z
- 参考现有实现 ABC
```

### 上下文传递

必须传递的信息：
- 项目根目录路径
- 相关文件位置
- 技术栈和依赖版本
- 现有代码风格/规范
- 特殊约束条件

## 特殊情况处理

### 任务中途发现比预期复杂

**立即切换策略**：
1. 停止当前直接编写
2. 告知用户任务复杂度超出预期
3. 建议改用 Kimi Code 处理
4. Spawn 子代理继续

### Kimi Code 执行失败

**故障排查流程**：
1. 查看子代理输出和错误日志
2. 分析失败原因（依赖？权限？理解偏差？）
3. 修复环境或澄清需求
4. 重新 spawn 或调整策略

### 用户要求"简单改一下"但涉及多文件

**坚持评估原则**：
- 即使用户说"简单"，也要客观评估
- 礼貌说明："这个改动涉及 X 个文件，建议用专业工具确保质量"
- 给出两种方案让用户选择

## 工作检查清单

### 处理编程任务前

- [ ] 是否充分理解需求？
- [ ] 是否评估了任务复杂度？
- [ ] 是否选择了合适的执行模式？
- [ ] 复杂任务是否准备了完整的上下文？

### Spawn Kimi Code 前

- [ ] 任务描述是否清晰完整？
- [ ] 项目路径是否正确？
- [ ] 是否说明了验收标准？
- [ ] 是否提供了必要的约束信息？

### 任务完成后

- [ ] 是否验证了结果正确性？
- [ ] 是否向用户清晰汇报了改动？
- [ ] 是否说明了使用方式和注意事项？
- [ ] 是否有需要后续跟进的事项？
